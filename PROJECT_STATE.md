# Поточний стан проєкту DripCAD / Drip Designer Pro

**Призначення цього файлу:** швидкий огляд структури та запуску після змін у середовищі (перейменування папки, новий ПК, інший Python). **На початку сесії** переглядайте його, **наприкінці** — оновлюйте зміст і дату внизу. Рішення з роботи з AI теж коротко фіксуйте тут; код зберігайте на диск (`Ctrl+S`), за можливості робіть commit — так контекст не залежить лише від історії чату.

**Репозиторій:** підключено до **GitHub** (`origin`); перед push — `git pull` / вирішення конфліктів за потреби.

## Що це за проєкт

Desktop-додаток на **Python + Tkinter** для проєктування крапельного зрошення: креслення поля (у т.ч. кілька окремих **блоків**), сабмейни, авто- та ручні латералі, гідравлічний розрахунок (Hazen–Williams), рельєф/ізолінії, локальні SRTM-тайли та експорт KML/DXF/PDF. Окремо від основного вікна (меню **Інструменти**) — польовий калькулятор латераля та підбір телескопа сабмейну за бюджетом напору. Детальніше — у [PROJECT_CONTEXT.md](PROJECT_CONTEXT.md).

## Точка входу (як запускати)

З кореня репозиторію:

```text
py main_app/main.py
```

або

```text
python -m main_app.main
```

У `main_app/main.py` корінь проєкту додається в `sys.path`; **робочі шляхи до файлів** задаються через `main_app/paths.py` (`PROJECT_ROOT`, `PIPES_DB_PATH`, `DESIGNS_DIR`, `SRTM_DIR`) — **не** через `os.chdir`, щоб не залежати від `cwd` IDE.

**Важливо про Python на Windows:** якщо команда `python` вказує на MSYS/MINGW без `pip` і без `shapely`, використовуйте **`py`** (Windows Python Launcher) — зазвичай там уже є потрібні пакети (`shapely`, `fpdf2`, `Pillow`).

## Дані та проєкти

- `designs/<Назва_проєкту>/` — JSON проєкту (`<Назва>.json`) + копія `pipes_db.json` у **ті самій** теці. Усі стандартні збереження та **«Зберегти проект як»** за замовчуванням потрапляють сюди (не на довільний шлях з діалогу). Ім’я теки/файлу санітизується під обмеження Windows.
- `pipes_db.json` у корені — глобальна база труб (шаблон).
- `_srtm_/` — кеш повних файлів `*.hgt` (розпаковані); у `.gitignore` ігнорується вміст, лишається `_srtm_/.gitkeep`.

**JSON проєкту:** окрім `points`, `geo_ref`, `topo_points`, `calc_results`, `params`, `allowed_pipes`:

- **`field_blocks_data`** — список блоків поля; кожен елемент містить `ring`, опційно `edge_angle`, `submain` (полілінії сабмейнів), `auto` / `manual` (координати латералів), **`params`** (словник знімка налаштувань блоку: крок ліній/емітерів, max довжина, групування рядів, опційно поля емітерів). У старих файлах без `params` використовується fallback на глобальні поля після завантаження.
- **`scene_lines`** — список декоративних поліліній (локальні XY), ескіз/ситуація; не використовується в розрахунках.
- **`trunk_map_nodes`** / **`trunk_map_segments`** — магістраль на карті/полотні: вузли (`kind`: source, valve, bend, junction, consumption; координати `x`,`y`, опційно `lat`,`lon`; у споживачів опційно **`trunk_schedule_q_m3h`**, **`trunk_schedule_h_m`** для сценарію «магістраль за поливами»), сегменти як **ребра** (два індекси в `node_indices` + **`path_local`** — полілінія труби в метрах; опційно **`pipe_material`**, **`pipe_pn`**, **`pipe_od`** після діалогу труби). Після завантаження/завершення траси нормалізація до пар ребер у `file_io_impl.py` / `DripCAD.normalize_trunk_segments_to_graph_edges`.
- **`field_blocks`** — лише кільця для сумісності; повне відновлення — з `field_blocks_data`.

## Поведінка UI (стисло)

### Режими креслення (полотно «Без карти» та вкладка «Карта»)

Перемикачі режимів і **ADD/DEL** на вкладці **«Карта»** — у **нижній частині лівої колонки**, у **панелі малювання** → вкладка **«Малювання»** (синхронізація з `mode` / `action`). Верхня частина лівої колонки — **зона / тайли / шари** (розділювач `PanedWindow`). Поле **L (м)** з **Enter** (`add_by_length`) — у **низу правої панелі** (`control_panel_impl.py`).

1. **VIEW** — простий перегляд (без редагування); режим за замовчуванням після старту та після відкриття проєкту.  
2. **DRAW** — контур блоку; ПКМ замикає.  
3. **SET_DIR** — ЛКМ у блоці, потім два ЛКМ напрямку рядів → авто-латералі цього блоку. Якщо на карті задана **зона проєкту**, після **«ОНОВИТИ СІТКУ»** ці лінії тягнуться на **всю зону проєкту** (AABB), а не обрізаються лише контуром блоку; без зони — обрізання по полігону блоку.  
4. **SUBMAIN** — сабмейн у блоці; snap до перетинів і **кінців** латералів уздовж **променя** у напрямку курсора (клавіша **S** — увімк/вимк snap).  
5. **DRAW_LAT** — ручна dripline; ПКМ — фіналізація з підрізанням/подовженням до сабмейну.  
6. **PAN** — панорама.  
7. **INFO** — після розрахунку: **ЛКМ по латералі** — графік ряду; **ЛКМ по сабмейну** — **H, Q та рельєф (ΔZ від початку)** вздовж магістралі (`submain_profiles`).  
8. **RULER** — лінійка.  
9. **CUT_LATS** — лінія зрізу авто/ручних латералів.  
10. **SUB_LABEL** — розміщення підписів секцій сабмейну (ЛКМ); **не** скидає повний `reset_calc` при кожному кліку там, де це не потрібно.

11. **LAT_TIP** — оцінка тиску на тупиках латераля (поле).

Кнопка **«Завершити блоки → напрямок рядів»** перед першим SET_DIR для нових контурів. **Delete/BackSpace** — чернетка контуру або останній блок; окрема кнопка — очистити всі контури.

**Чернетка контуру та розрахунок:** під час **DRAW** з незамкненим контуром ЛКМ **не** викликає `reset_calc()`; після **замикання** нового блоку (`close_polygon`) гідравліка також **не** скидається автоматично — лише коли зміни справді інвалідують результати.

### Гідравліка та креслення після розрахунку

- У розрахунок передаються **індекси блоків** по кожному сабмейну (`submain_block_idx`) і **індекс блоку на латераль** (`lateral_block_idx`); у ядро можуть йти **`e_steps` / `e_flows`** (крок і витрата емітера по кожному латералю з `params` блоку або fallback з панелі). На вкладці **«Гідравліка»** дроплісти матеріалу/PN **прибрані** з панелі: рядок-статус + кнопка відкриття вікна **вибору труб проєкту** (там же mat/PN); приховані combobox’и лишаються для сумісності з `load_project` і трейсами.
- **Фіксована к-сть секцій / кількість секцій / H на крані / «підганяти сабмейн під тиск»:** зміна цих елементів **не** викликає глобальний `reset_calc` по всіх блоках — інвалідується лише гідравліка **активного** блоку (`dripcad_legacy._invalidate_hydro_ui_active_block_or_all`); якщо блоків немає — повне скидання як раніше. Команда чекбокса «фіксована к-сть» лише вмикає/вимикає поле кількості (без подвійного скидання).
- **Малювання нового блоку:** ЛКМ у режимі `DRAW` більше не проходить через загальну гілку глобального `reset_calc`; додавання нового контуру не стирає гідравліку вже порахованих блоків.
- На вкладці гідравліки — **«Скинути»** по номеру блоку (`clear_hydro_block`): вирізаються секції/емітери цього блоку, клапани за прив’язкою до сабмейнів блоку, очищаються `section_label_pos` та **submain_profiles**.
- **Колір секцій на полотні:** при малюванні береться з **поточного `pipe_db`** за mat/PN/Ø (`dripcad_legacy._section_draw_color`), а не лише з застиглого `calc_results`, щоб зміна кольору в базі одразу відображалась після перемальовування.
- **Відображення секцій:** суміжні відрізки з однаковими mat/pn/d на одному сабмейні **зливаються** для креслення; полілінія **розбивається за поворотами** (~≥22°) — підписи **L** пропорційні геометрії шматка (одна гідравлічна секція може дати кілька підписів на прямих після кута). Ключі позицій підписів: `"{sectionIndex}_{subIdx}"` та fallback для `sub_idx==0`.
- **Чому після повороту на кресленні/у списку може змінитись d:** фізичні розрізи завжди проходять через **вершини** полілінії сабмейну; діаметр відрізка береться з **математичної зони** за **серединою** відрізка (різна Q вздовж магістралі) — деталі в коментарі в `hydraulics_core.py` біля `final_cuts`.
- **Режим `SUB_LABEL`:** перетягування підпису стало двокроковим: 1-й ЛКМ — вибір найближчого підпису (він «прилипає» до курсора), 2-й ЛКМ — фіксація нового місця; тимчасовий стан скидається через `reset_temp()`.
- **Збереження позицій підписів:** після повторного розрахунку вручну пересунуті підписи відновлюються для тих же секцій (`sm_idx + section_index + sub_idx`), без повернення на авто-позицію.
- **Колізії підписів між сабмейнами:** ключ `section_label_pos` включає `sm_idx`, тому однакові `section_index` у різних сабмейнах більше не конфліктують.
- **Звіт гідравліки:** окрема вкладка **«Результати»** з дроплістом блоку; popup-вікно «Специфікація та Гідравліка» прибрано. Для блоку показується **площа поля** за контуром `ring` (м² та га).

### Вкладка «Блок»: авто-латералі та крапельниці на мапі

- Фільтр **кожну N-ту** задає лише видимість авто-латералей після розрахунку.
- Фільтри **номер від крана** та **номер з кінця** працюють тільки коли **кожну N-ту** вимкнено; вмикання будь-якого з них автоматично вимикає **кожну N-ту**.
- **Текстовий перелік** крапельниць активного блоку з **H** поза діапазоном (мін/макс з «Гідравліка», допуск ±0,02 м як у розрахунку); нумерація латераля — глобальний порядок.
- **Маски переливу / недоливу на мапі (контури):** суцільні зовнішні контури зон без «крапель» усередині; геометрія — union буферів навколо проблемних емітераів + buffer між рядами; **перелив** додатково **зшивається з ізолінією Q = Q ном** (`var_emit_flow`) у межах блоку. Під час увімкнених масок **приховані** ізолінії «Діаграма виливу» на полотні; латералі **активного** блоку без червоного/жовтого аудиту (лише контури зон). Кеш Shapely-результату між перемальовуваннями + спрощення вершин для швидкого Canvas.
- **Крапельниці (вилив) — старий режим ізоліній:** якщо маски вимкнені, «Діаграма виливу» будує IDW-ізолінії по полю **q_emit** (може виглядати як щільна сітка точок); колір сегмента за класифікацією **H** (норма / перелив / недолив).
- Задано діапазон тиску на крапельниці: у межах діапазону — **один колір** «норма» на графіках виливу; градієнт за Q — лише **поза** діапазоном (де застосовується).

## Найближчий план

### A. Поточний фундамент (блоки, BOM, редактор сабмейну)

**Модель проєкту**

- Проєкт = набір **блоків** поля; у JSON уже є `field_blocks_data[]` з `ring`, `submain`, `auto`/`manual`, **`params`** (знімок налаштувань блоку).
- **BOM** за задумом складається з:
  - BOM **кожного блоку** (латералі / крапельниці / труби блоку);
  - окремо BOM **магістралі** (сабмейн(и), арматура, труби магістралі) — у перспективі узгодити з окремою моделлю магістралі (п. B).
- Кожен блок може мати **свою** схему поливу, напрямок рядів, параметри крапельної лінії — це вже частково в `params`; потрібна узгодженість UI ↔ збереження ↔ розрахунок **в контексті обраного блоку**.

**UI вибору активного блоку**

- Дропліст **«поточний блок для проєктних робіт»** по центру панелі меню **або** окрема верхня смуга.
- Усі дії креслення/редагування, релевантні одному блоку (напрямок рядів, прив’язка сабмейну, фільтри відображення тощо), застосовуються до **обраного** блоку (де вже є `block_idx` — розширити покриття).

**Редактор сабмейну (крок паралельно з п. B)**

- Окремий режим або діалог: список **сегментів** полілінії сабмейну (довжина, мат/PN/Ø, кут, порядок).
- **Ручна корекція довжини** кожного сегмента з прив’язкою до **стандартної довжини труби** з `pipes_db` (кратність цілим числу «штанг» або кроку з каталогу; валідація сумарної довжини / координат вершин).
- Зв’язок із гідравлікою: після правок — оновлення геометрії в `field_blocks_data` і скидання/перерахунок залежних результатів.

### B. Наступний спроектований етап — **магістраль** (відкритий граф-дерево)

**Концепція**

- **Магістраль** моделюється як **дерево** (відкритий зв’язний граф без циклів), орієнтований від **витоку**:
  - **Витік (корінь)** — точка підключення до джерела / насосної станції / напору на вході мережі (задані **H₀**, **Q** або режим насоса — уточнити на етапі DTO).
  - **Точки повороту** — вершини лише зі зміною напрямку труби (топологія + геометрія на карті / у локальних координатах).
  - **Точки розгалуження** — вузли степеня ≥3: один вхідний напрямок (від батька) і кілька виходів на дочірні ділянки.
  - **Точки споживання (листя)** — підключення **навантажень**: блок(и) поля, зони поливу, клапани або агреговані вузли з відомим/заданим **графіком споживання**.
- **Ребра** — ділянки трубопроводу між вершинами: довжина, діаметр/матеріал (з `pipes_db`), опційно профіль рельєфу вздовж ділянки.
- **Графік споживання** — прив’язаний до вузла споживання (або до групи блоків): часовий ряд **Q(t)** (м³/год, л/с) або набір кроків / добовий шаблон; для розрахунку — узгодження з існуючою гідравлікою блоків (пікове Q, тривалість зон, послідовність відкриття).

**Етапи реалізації (чернетка)**

1. **Дані:** JSON/DTO для графа магістралі (`nodes[]`: type, coords, payload; `edges[]`: u, v, pipe spec, L); окремо `consumption_profiles[]` з посиланнями з листя.
2. **UI:** режим креслення / таблиця вершин; валідація дерева (один корінь, без циклів); імпорт з існуючих поліліній сабмейну як чернетка топології (опційно).
3. **Відображення:** шар на canvas поверх/поруч з блоками; підписи вузлів і напрямок від витоку.
4. **Гідравліка:** окремий розв’язувач HW по дереву з граничними умовами на листі з Q(t) або зведеним Q_max; згодом — зв’язка «листо ↔ блок field_blocks» для узгодженого Q.
5. **Звіт / BOM:** довжини і типи труб по ребрах; арматура на розгалуженнях.

**Ризики / рішення**

- Поточні **сабмейни в блоці** залишаються робочою моделлю; магістраль вводиться **паралельно** і з часом з’єднується (наприклад, лист споживання = «підключити блок N» або «підсумок сабмейнів блоку»).

**Довгостроково (після п. A–B)** — див. нижче «Що логічно доробити далі».
