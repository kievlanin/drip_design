# План: профіль магістралі, кольори латералів за Q, панель «Блок»

**Статус:** чернетка плану до реалізації (оновлюйте цей файл після виконання пунктів).

## 1. Профіль прокладки (обрізається до «насос — перший пікет»)

**Причина в коді:** у `main_app/ui/dripcad_legacy.py` функція `_trunk_segment_world_path` при `len(node_indices) == 2` одразу повертає відрізок між двома вузлами і **не підставляє** `path_local`, хоча для сегментів з кривою/пікетами повна траєкторія часто зберігається саме там.

**Зміна:** після побудови двовузлового шляху порівняти з валідним `path_local` (той самий парсер полілінії): якщо `path_local` має ≥3 точок **або** сумарна довжина полілінії по `path_local` помітно більша за довжину хорди між двома вузлами — **використовувати `path_local`**. Інакше залишати відрізок між вузлами (зворотна сумісність для справді прямих сегментів). Допоміжно можна перевірити `_polyline_length_m` для обох варіантів.

`open_trunk_segment_ground_profile` змінювати не обов’язково — воно вже дискретизує весь `path`, який поверне оновлена `_trunk_segment_world_path`.

## 2. Min/max вилив по латералю та порівняння з номіналом ±5%

**Джерело фактичних Q:** у розрахунку вже є `calc_results["emitters"]` з ключами `lat_{idx}` і рядками крил `L1`/`L2` з полем `q_emit` (л/год) — та сама структура, що використовується для аудиту тиску в `modules/hydraulic_module/hydraulics_core.py`.

**Номінал Q по латералю:** узгодити з UI — як у `_per_lateral_emit_steps_flows` у `dripcad_legacy.py`: для кожного глобального індексу латераля `params.flow` блоку або fallback `var_emit_flow`. Ці ж `e_flows` уже передаються в гідравліку при зборі даних.

**Де рахувати:** у тому ж місці в `hydraulics_core.py`, де формується `lateral_pressure_audit`: для кожного `lat_{k}` зібрати всі `q_emit > 0` з `L1`+`L2`, обчислити `q_min`, `q_max`, взяти `q_nom` з масиву `e_flows[k]`, порівняти з допуском **5%**:

- `q_max > q_nom * 1.05` → статус **overflow_q** (червоний на мапі);
- інакше якщо `q_min < q_nom * 0.95` → **underflow_q** (охра, наприклад `#CC7722` або узгодити з наявним `#E8C547`);
- інакше якщо обидва в межах **[0.95·q_nom, 1.05·q_nom]** → **ok_q** (зелений).

**Одночасне порушення** (макс вище +5% і мін нижче −5%): зафіксувати одне правило — наприклад **пріоритет червоного** або окремий колір «обидва» як у тисковому аудиті (`#FF6600`).

Записати в `calc_results` окремий словник, напр. `lateral_flow_audit` з ключами `lat_{k}` і полями: `q_min`, `q_max`, `q_nom`, ідентифікація емітерів min/max (крило `L1`/`L2`, `x` вздовж латераля).

## 3. Малювання латералів на полотні

У `dripcad_legacy.py` у `_draw_lateral_with_audit` / `_lat_line_color` зараз кольори від **тиску** (`lateral_pressure_audit`).

**Зміна:** якщо є запис у `lateral_flow_audit` для цього `lat_{li}` і є еміттери з Q — застосувати **спочатку** правила кольору за виливом; якщо запису немає — залишити поточну логіку тиску. Для простоти — **один колір на весь латераль** за глобальним min/max Q.

## 4. Вкладка «Блок»: текст з крапельницями max/min Q

- У `control_panel_impl.py` додати окремий блок (наприклад `LabelFrame` під існуючим «Крапельниці поза діапазоном тиску») з `ScrolledText` або розширити оновлення панелі.
- У `refresh_block_out_of_range_emitters_panel` (або новій функції) заповнювати список **лише для латералів активного блоку** з даних `lateral_flow_audit`.

Виклик оновлення — там же, де вже викликається `refresh_block_out_of_range_emitters_panel` після розрахунку / зміни активного блоку.

## 5. Тести

- Юніт-тест на `_trunk_segment_world_path`: сегмент з `node_indices` [0,1] і довшим `path_local` — очікується довша полілінія.
- За можливості — тест на побудову `lateral_flow_audit` з мок-емітерами.

## Трека робіт (коротко)

1. `_trunk_segment_world_path`: при 2 `node_indices` підставляти `path_local`, якщо він довший/детальніший.
2. `hydraulics_core`: `lateral_flow_audit` (q_min/q_max/q_nom + ідентифікація емітерів).
3. `dripcad_legacy`: кольори латералів за `lateral_flow_audit` (+ fallback на тиск).
4. `control_panel_impl` + refresh: блок тексту з min/max Q по латералях активного блоку.
5. Тести на path та/або flow audit.
